# Resolution rules & safety gates

Read just-in-time before any write. This skill mutates GitHub — bias hard toward
**skip** when uncertain.

## What the GitHub MCP can and cannot do (important)
The github-agent MCP exposes read/triage/verify and **close** (via
`github_issues action=update` / `github_pulls action=update` with `{"state":"closed"}`).
It has a native merge (`github_pulls action=merge`, or the standalone REST
`github_merge_pull_request`) AND a native comment tool (`github_comments`,
create/list/get/update/delete). So:
- **triage / verify / close / comment / merge** → github-agent MCP tools (native
  names; under the multiplexer they carry the `gith__` prefix): comment via
  `github_comments action=create`, merge via `github_merge_pull_request`.
- **Fallback** — `scripts/gh_write.py` (GitHub API helper; same token as git push):
  use it only if the deployed github-mcp predates `github_comments` (rebuild/redeploy
  to get it), or when you want its atomic re-check-`mergeable_state==clean`-then-
  comment-then-merge in a single guarded step.

## Verify BEFORE deciding (does the work still exist?)
- **Issue:** confirm the problem is real and still present, OR already fixed.
  Gather **evidence**: a merged PR / commit that addresses it
  (`github_search action=issues`, `github_commits action=list`, `github_contents
  action=get`, `github_search action=code`). An issue is `safe_close` ONLY with cited
  `resolved_evidence`. No evidence → **skip** (human triage). Never close an unverified issue.
- **PR:** `github_pulls action=get` for `mergeable_state`, `additions/deletions`,
  review state; `github_actions action=list_runs` (branch = PR head) for checks.

## Verdicts (see `classify_safe.py` — it encodes these deterministically)
- **safe_merge** — ALL of: not draft; `mergeable_state == clean`; all checks
  `success`; AND in an auto-merge **allow-class**. Default allow-class:
  `dependabot`/`renovate` **patch or minor** bumps. MAJOR bumps, security-sensitive,
  core, or large diffs are **never** auto-merge — skip to a human. A human-approved
  PR may be allowed with `allow_class=approved`.
- **safe_close** — PR: `superseded_by` a merged PR, OR stale (`age_days` >
  `--stale-days`, default 60) **and** `dirty` (conflicts) = abandoned. Issue: only
  with `resolved_evidence`.
- **skip** — everything else; record the blocking reason.

## Act (gated)
1. **Dry-run first.** Produce the plan (verdict + action + draft comment per item)
   and show it. Nothing writes without explicit confirmation, or a pre-agreed
   autonomy level the user set for this run.
2. **Comment, then act.** Always post an explanatory comment before/with the action.
   Every comment MUST start with the disclaimer:
   ```
   > *This was generated by AI during triage.*
   ```
3. **Close** via `github_issues action=update` / `github_pulls action=update`
   (state=closed). **Merge** via
   `gh_write.py merge --confirm` (it re-checks `mergeable_state==clean` at merge time
   and refuses drafts/non-clean). **Comment** via `gh_write.py comment --confirm`.
4. **Re-check at write time** — state drifts between triage and action; the merge
   helper re-fetches the PR and refuses if it is no longer clean.

## Hard "never"s
- Never merge with failing/pending required checks or non-`clean` mergeable state.
- Never force-merge, never override branch protection, never merge a `major` bump automatically.
- Never close an issue without cited resolution evidence.
- Never bulk-act without a dry-run the user has seen, unless they set an explicit autonomy level.

## Learn from outcomes (optional)
After acting, record the result with `graph_feedback` (under the multiplexer,
`go__graph_feedback`)
(`correction_type=action_outcome`, `target_id=<repo>#<num>`,
`corrected_value={"success":true/false,...}`) so the playbook prefers actions that
turned out right.
